# RK3588 Factory Device Communication Hub — 最终设计定稿

> 本文档是项目的完整设计文档。所有模块全部定稿后开始实现。

---

## 一、系统定位

**工业现场设备信息传输中枢 + 协议转换中间件**

接收来自不同产线设备（PLC/传感器/机器人/AGV/扫码枪）的数据，完成协议解析、消息路由、规则判断、持久化存储、实时监控全流程。

│         │ shm_ring_write               shm_ring_read               │

│  │  protocol_engine (协议解析引擎)                                │   │

│  │  · 调用插件链：先经 raw_tcp 拆包 → 再经 modbus/can 解析      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│         │ pipe                                                      │
│         ▼                                                           │

│  │  · WAL 模式 + 批量提交（10条/5秒）                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  monitor (系统看门狗)                                         │   │

│  │  · 故障三级恢复：心跳超时→SIGTERM→SIGKILL→重启               │   │
│  │  · 对外暴露 /proc 风格的进程资源视图                           │   │

│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  http_server (手写 HTTP 状态 API)   监听 0.0.0.0:8080         │   │
│  │  GET  /status      → 系统/进程/设备概要                        │   │
│  │  GET  /devices     → 完整设备列表（分组+状态+最新数据）        │   │
│  │  GET  /devices/:id → 单个设备详情+追踪+延迟                    │   │
│  │  GET  /alarms      → 当前告警列表                              │   │
│  │  GET  /trends      → 设备数据趋势（最近 N 条）                 │   │
│  └─────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────┘

### 2.2 进程间接口

| 生产者 | 消费者 | IPC 方式 | 数据类型 |
|--------|--------|---------|---------|
| gateway | router | 消息队列（System V） | 原始 TCP 字节流 |
| router | protocol_engine | 共享内存（环形缓冲区） | 原始帧 (uint8_t[]) |
| protocol_engine | router | 共享内存（环形缓冲区） | 解码后的 protocol_frame_t |
| router → storage | — | 命名管道（FIFO） | 序列化后的消息记录 |
| monitor | 所有进程 | 共享内存（只读） | 心跳时间戳 + 统计信息 |
| console | router/storage | 命名管道 | CLI 命令/回复 |
| http_server | router/monitor/storage | 共享内存（只读） | JSON 字符串 |

## 五、面试价值总结

### 面试官会问你的 10 个问题

```
Q1: epoll 的 LT 和 ET 有什么区别？你为什么选 LT？
Q2: 共享内存的 volatile 在 ARM 多核上够用吗？什么时候加屏障？
Q3: 你的路由表怎么实现？100 台设备 O(1) 查找？
Q4: 规则引擎怎么设计的？如果并发 10 条规则同时匹配怎么办？
Q5: 分布式追踪怎么保证不影响正常消息的处理延迟？
Q6: Modbus 协议有哪些功能码？你解析了哪几个？
Q7: 插件 dlopen 有什么安全问题？怎么隔离？
Q8: 压测里环形缓冲区溢出了，你怎么发现的？怎么改进？
Q9: 进程 top 的数据来源是哪里？/proc 里的每个字段是什么意思？
Q10: 这个系统如果在 100 台设备的产线上跑，哪里最先到达瓶颈？
```
